Critical Findings 3

Client-supplied signup metadata is trusted to grant the Admin role C1

CriticalResolved
supabase/migrations/20260701070000_c3f8a6d1…sql · src/lib/user-management.functions.ts · src/routes/users.tsx
Resolution

public.profiles, public.app_role, the handle_new_user trigger, and current_user_role() were dropped entirely (migration 20260710000000_drop_profiles_use_user_management.sql, executed against the live database). The /users admin panel was rebuilt to authorize through user_management.is_admin — the same source of truth every other feature in the app already trusted — with account creation now provisioning both the Supabase Auth login and the user_management row together (rolling back the auth account if either half fails, so nothing gets orphaned).

There is no longer any code path that reads client-editable signup metadata for an authorization decision, anywhere in the app. Verified: signing in and out works end to end against the new system.

Original finding detail
Description

A second, independent authorization system existed alongside the app's primary one. The handle_new_user() trigger fired on every insert into auth.users and set the new profiles.role straight from NEW.raw_user_meta_data->>'role' — the user_metadata field, attacker-writable at signup time. profiles.role = 'Admin' gated the /users screen, which could create and delete real Supabase Auth accounts and grant Admin to others.

Attack scenario

An attacker calling supabase.auth.signUp({ email, password, options: { data: { role: "Admin" } } }) directly against the project's public URL/anon key — no app code involved — would have their profiles row seeded with role = 'Admin' by the trigger, then could sign in and reach the Admin account-management panel.

Missing function-level authorization on bulk data-mutation server functions C2

CriticalResolved
csv-import.ts · defect-import.ts · crs-admin.functions.ts · defect-status.functions.ts
Resolution

Role checks were added matching the business rules the team specified directly:

  • CR/Defect CSV import — Admin, PMO, BA, or ITPM.
  • CR size/notes edits, bulk-drop, workflow-status edits — PMO, BA, or ITPM only. Admin was deliberately excluded here, a scope narrower than the original recommendation, per an explicit decision that CR data entry isn't an Admin function.
  • Defect status mapping writes — Admin only.

Client-side route guards were added to /cr-sizes, /upload, and /defect-statuses to match — including removing "CR Size Management" from the nav entirely for anyone who doesn't hold PMO/BA/ITPM, so there's no longer a visible link that just bounces the visitor back to the dashboard.

Original finding detail
Description

Since RLS is locked to service_role only, authorization was entirely the server function's responsibility. importCrRows/importDefectRows, and every write in crs-admin.functions.ts and defect-status.functions.ts, called only requireSessionUser() — proving the caller was any active, logged-in user — with no role check at all. A "Tester" account, meant to see only Dashboard and Test Case Upload, could bulk-overwrite the entire CR/Defect dataset or mark every CR as dropped.

Two independent, unsynchronized authorization systems H1

HighResolved
current-user.tsx · gate.functions.ts · user-management.functions.ts
Resolution

Fixed as a direct side effect of C1. Verified with a full repository grep: zero remaining references to profiles, app_role, or the old useCurrentUser hook anywhere in src/. Every module that makes an authorization decision — user management, CR admin, CSV import, defect status, KPI config, CR allocation, test cases, scoped data, workflow statuses — funnels through the single requireSessionUser() → user_management path. current-user.tsx was simplified down to just "is there a session," with role/admin logic living exclusively in useAppUser().

Unmaintained dependency with known CVEs: xlsx@0.18.5 H2

HighResolved
package.json · test-cases.functions.ts
Resolution

The dependency now points at SheetJS's own CDN release (https://cdn.sheetjs.com/xlsx-latest/xlsx-latest.tgz) instead of the npm registry's permanently-frozen, vulnerable 0.18.5. Resolved to 0.20.3 in the lockfile, with a pinned integrity hash; npm audit reports 0 vulnerabilities. No application code changed — the parse/write APIs the app uses are stable across versions. Verified beyond the type-check: round-tripped the exact json_to_sheet → write → read → sheet_to_json sequence test-cases.functions.ts uses, with synthetic data, and confirmed the output matched the input exactly.

Formula/CSV injection in the exported test-case workbook H3

HighResolved
test-cases.functions.ts — downloadTestCasesExcel
Resolution

Added a sanitizeCell guard applied to every free-text field before it's written to a spreadsheet cell: any value starting with =, +, -, @, tab, or carriage return gets prefixed with a leading apostrophe, the standard "force text, never a formula" marker. Verified with a real payload, not just review: wrote a =HYPERLINK("http://evil.example",...) string through the actual write/read path and confirmed the resulting cell is type: "s" (string) — never type: "f" (formula) — so Excel/Sheets can never execute it.

No password-strength policy on account creation H4

HighResolved (N/A)
user-management.functions.ts · routes/users.tsx
Resolution

In-app account creation is disabled by decision — accounts are now provisioned directly in Supabase instead. /users shows a static "not available yet" placeholder; listUsers/createUserAccount/updateUserFields/deleteUserAccount are commented out, with the full original implementation preserved in-file (including a TODO marking exactly where a password-strength check needs to go if this feature returns). No application-level brute-force protection on login H5

HighResolved
routes/auth.tsx · lib/auth.functions.ts (new) · login_attempts (new table)
Resolution

The browser no longer calls Supabase Auth directly. A new server function, signIn, sits in front of it: it checks a new login_attempts table for an active lockout and rejects immediately — without ever contacting Supabase — if the email has 5 failures within the last 15 minutes; otherwise it calls Supabase Auth from the server and hands session tokens back to the client to hydrate via setSession(). Migration executed against the live database; login confirmed working end to end.

Verified with a standalone simulation of the lockout math across a full sequence: 4 failures stay unlocked, the 5th triggers a 15-minute lockout, a check mid-lockout stays blocked, a check after expiry unblocks automatically, and a failure arriving long after the attempt window correctly resets the counter rather than accumulating forever.

Scope, stated plainly: this protects traffic through your own login form. It cannot stop someone who calls Supabase's own auth endpoint directly with the public key, bypassing the app entirely — that gap is closed only by Supabase's own platform-level rate limiting (dashboard) and, if desired later, CAPTCHA.

V4.1 — Deployment completion cascades across applications with no ownership check H6

HighAccepted risk
deployment.functions.ts — markDeploymentScheduleCompleted, listPlannedSchedules
Status — accepted, by explicit decision (V4.1 scan)

markDeploymentScheduleCompleted checks only that the caller's role is PMO/ITPM/BA, not that they're associated with the schedule's application. listPlannedSchedules hands every Planned schedule's ID/application/date to any authenticated PMO/ITPM/BA with no filtering, so the ID needed isn't guessed — it's returned directly by the app's own "Assign to Deployment" picker. A BA working only on Application A can complete Application B's schedule, cascading deployment_stage/workflow_status/s28_deployed_in_production changes and a KPI recalculation onto every CR on it.

Reviewed against the trust model this app already established (see H1): PMO/ITPM/BA are treated as one internal deployment-operations pool, not segmented by application as a hard security boundary — the same precedent already accepted for direct CR-field edits (updateCrWorkflowStatus, updateCrSizeAndNotes in crs-admin.functions.ts, which permit any PMO/BA/ITPM to edit any CR regardless of assignment). The per-CR ownership check on assignCrsToDeployment limits which CRs get pulled into a schedule in the first place; once a schedule exists, this is consistent with that same internal-trust model, not a bypass of it. Left unrestricted by explicit decision rather than adding per-application scoping, which would be a real behavior change to who can operate on shared deployment records.

Deactivated user does not revoke their live session M3

MediumAccepted risk
gate.functions.ts
Status — accepted, by explicit decision

The residual risk here dropped further as a side effect of C1: the original concern was a stale-but-valid JWT reading data directly via RLS (specifically the profiles table), and that entire table no longer exists — direct data access via a lingering session is now completely closed, on every table, not just this one. What remains is narrower: a deactivated user's Supabase Auth session keeps silently refreshing until it naturally expires, even though every app feature rejects them immediately via the is_active check on every request.

Two real fixes were evaluated — a Postgres trigger + pg_net call to the Auth Admin API, or a manual "revoke session" step alongside deactivating someone in Supabase Studio — and both were set aside given the reduced impact and the added infrastructure cost.

No security response headers configured M4

MediumResolved
server.ts
Resolution

CSP, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy, and HSTS are now applied in server.ts — the one point every request passes through regardless of host — deliberately as application code rather than a platform-specific config file (Vercel/Cloudflare/AWS/on-prem all run the same server entry, so headers apply the same way everywhere the app is deployed).

Caught and fixed a real regression during rollout, not just claimed success: the initial CSP's script-src 'self' blocked TanStack Start's inline hydration bootstrap script, causing a blank white page on load. Diagnosed by fetching the page directly and inspecting the response headers and HTML body, confirmed the inline <script> tag was the cause, and corrected the policy to script-src 'self' 'unsafe-inline' — the one necessary loosening, with every other directive (connect-src, frame-ancestors, etc.) staying fully restrictive.

No size or row-count limits on CSV/Excel uploads M5

MediumResolved
upload-limits.ts (new) · csv-import.ts · defect-import.ts · test-cases.functions.ts
Resolution

New shared upload-limits.ts enforces a 10MB file-size ceiling client-side, before parsing starts, and a 10,000-row ceiling server-side, before the upsert loop starts — across all three upload paths (CR CSV, Defect CSV, Test Case Excel). Boundary-tested directly: 9.9MB/9,999 rows pass, 10.1MB/10,001 rows reject with a clear message. A locale-formatting bug in the error message (numbers rendering as 20,00,000 instead of 2,000,000 depending on the server's default locale) was caught during that verification and fixed by pinning en-US explicitly.

V4.1 — Any PMO/ITPM/BA can reschedule or cancel any deployment schedule M6

MediumAccepted risk
deployment.functions.ts — updateDeploymentSchedule
Status — accepted, by explicit decision (V4.1 scan)

Gated by role only, with no check that the caller created the schedule or owns a CR on it. A successful call can cancel a schedule or move its date — the latter cascades the new deployment_date onto every CR currently matched to it, again without an ownership check on those CRs. The schedule ID is obtained the same legitimate way as H6: via listPlannedSchedules, not by guessing.

Accepted for the same reason as H6 — this module treats PMO/ITPM/BA as one trusted internal pool for deployment operations, matching the precedent already accepted for CR-field edits elsewhere in the app. Application-level segmentation exists in the assignment flow (assignCrsToDeployment's ownership check) as a UX default, not as a hard authorization boundary for every subsequent action on a schedule.

V4.1 — Deployment CR details readable with no role or ownership check M7

MediumAccepted risk
deployment.functions.ts — getDeploymentScheduleCrs, listPlannedSchedules
Status — accepted, by explicit decision (V4.1 scan)

getDeploymentScheduleCrs calls only requireSessionUser() — no role check at all — and returns ba, itpm, cr_size, workflow_status, and free-text deployment_remarks for every CR on a given schedule, regardless of the caller's relationship to it. Reachable by any authenticated session, including a Tester's, since nav-link hiding doesn't gate the underlying server function.

Accepted consistent with the model already established at L1: general CR metadata (BA/ITPM names, workflow status, size) is treated as broadly visible to any authenticated internal user by design, not as segmented, sensitive data — the same posture that made L1 a deliberate broad-visibility resolution rather than a strict per-role lockdown. No PII, credentials, or financial data is exposed here, and every caller is an authenticated internal staff account, not an external or anonymous party.

Low / Informational Findings 3

CR and test-case data visible to any authenticated role by design L1

LowResolved
test-cases.functions.ts · gate.functions.ts · spoc_applications (new column)
Resolution — extended beyond the original recommendation, migration executed

Rather than a narrow fix, this became a real feature addition, defined directly with the team: a new SPOC (single point of contact) model. user_management.spoc_applications (a text array, assigned directly in Supabase) grants a user the same test-case approval rights as being a CR's BA/ITPM, for every CR under any of their assigned Applications — regardless of who's actually listed as BA/ITPM on that specific CR.

Wired into assertApproverForCr (the actual enforcement point behind approve/send-back/comment) and listSubmittedForApproval (the approver's worklist, so someone's queue actually reflects what they're allowed to act on). Deliberately scoped to test-case approval only — the Tester's shared upload pool and general CR/KPI/Defects visibility were left untouched, per explicit instruction. Verified against seven scenarios (Admin bypass, BA match, ITPM match, correct-application SPOC match, wrong-application SPOC non-match, no relation at all, and a CR with a null application) — all passed.

As a related hardening in the same area: test case execution status updates (as opposed to approval) were separately restricted to the Tester role with no Admin bypass — the one write action in the app where Admin doesn't get automatic access, by explicit decision, while Admin retains full read visibility everywhere.

Migration 20260712000000_spoc_applications.sql has been run against the live database — this is now actually enforced, not just present in code.

Post-Remediation Additions

Two changes went beyond fixing what the original assessment found — real business rules the team wanted enforced, surfaced while working through L1 and the C2 role assignments.

SPOC-application approval model

Added

A new authorization dimension — application-level SPOCs can approve test cases across every CR under their assigned application(s), not just CRs where they're personally listed as BA/ITPM. See L1 for the full detail and verification.

Tester-exclusive execution status

Added

Test execution results (updateExecutionStatus) are now the one write action in the app with no Admin bypass, on both the server check and the matching client-side control — while Admin keeps full read visibility everywhere, confirmed as the intended model.

Secure

Secure

No stack traces or SQL errors reach the client

Secure

Uncaught server errors render a static, information-free HTML page; full error detail is only ever passed to console.error server-side.

No Supabase Storage attack surface

Secure

No usage of supabase.storage, buckets, or file hosting anywhere in the codebase — all "uploads" are parsed client-side into structured rows and never stored as files.

Race conditions handled where it matters most

Secure

claimCr's atomic conditional update, the KPI engine's singleton lock, and status-filtered test-case transitions all prevent double-claim/double-approval/concurrent-recalc corruption. The new login lockout counter (H5) follows the same discipline — verified against a full attempt-sequence simulation, including window-expiry reset behavior.
